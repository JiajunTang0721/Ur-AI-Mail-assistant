# Mail assistant

Mail assistant 是一个面向 Gmail 的智能邮箱整理助手。  
它不是简单地把邮件粗略分类，而是试图把“邮件应该展示在哪里”和“邮件本身有多重要”拆成两个独立维度，通过 **Gmail 原始数据 + 用户偏好 + LLM 特征抽取 + 本地优先级模型 + 用户反馈重训** 的完整链条，为用户提供可解释、可迭代的邮箱整理体验。

---

## 一、项目目标

本项目希望解决传统邮箱处理中的三个核心问题：

1. 用户每天会收到大量不同类型邮件，其中既有真正重要的内容，也有大量通知、订阅、推广和重复性邮件。
2. “需要优先看到”与“需要马上处理”并不是一回事，传统单维分类往往不能准确表达邮件的真实价值。
3. 用户对“什么邮件重要”的判断高度个性化，因此系统必须允许用户反馈持续修正模型，而不能只依赖静态规则。

因此，本项目的目标不是构建一个单纯的“邮件分类器”，而是构建一个 **可学习用户偏好的邮箱整理系统**。

---

## 二、系统核心思想

系统明确拆分两个不同维度：

### 1. 板块归属（Board Assignment）

表示这封邮件在前端界面中应该被展示到哪个板块：

- 优先内容
- 48h
- 待处理
- 可忽略
- 24h内已完成

这个维度强调的是 **展示语义 / 行动语义 / 查看时机语义**，而不是邮件本身的重要程度。

### 2. 优先级（Priority）

表示邮件本身的重要程度：

- high
- medium
- low
- ignore

这个维度主要用于：

- 邮件卡片颜色标识
- 同板块内部排序
- 用户心理优先级学习
- 后续模型训练优化

### 3. 为什么要解耦

如果把“优先内容 = 高优先级”“48h = 中优先级”写死，就会产生大量错误判断。  
例如：

- 一封 48h 内要查阅的邮件，可能只是 `medium`
- 一封待处理邮件，也可能是 `high`
- 一封宣传邮件可能内容很多，但本质仍然应被判为 `ignore`

所以本项目的核心设计原则是：

> **板块归属和优先级必须独立计算，不能强行绑定成一套标签。**

---

## 三、完整逻辑链条

整个系统的分析链路如下：

```text
Gmail 原始邮件获取
→ 结合项目已有设置与历史状态补充上下文
→ LLM 输出固定 schema 的结构化特征
→ 规则层 / 板块逻辑判断 board_type
→ 本地优先级模型输出 priority_score / priority_level
→ 写入本地数据存储
→ 前端按板块展示并按分数排序
→ 用户进行反馈
→ 重新分析时重训模型
```

这个流程的关键在于：

- **LLM 不直接做最终分类**
- **本地模型不直接决定板块**
- **用户反馈不会立即粗暴改板块，而是进入训练链路**

---

## 四、数据输入层

系统的输入不仅来自 Gmail API，还来自项目本身已有的本地状态。

### 1. Gmail 原始规范化字段

系统至少抽取以下字段：

- `message_id`
- `thread_id`
- `sender_name`
- `sender_email`
- `subject`
- `summary`
- `body_excerpt`
- `time_text`
- `timestamp`
- `is_unread`
- `has_attachment`
- `label_ids`
- `thread_message_count`

### 2. 项目内部补充上下文

系统还会把以下内容一起送入分析链路：

- 用户偏好设置
- 重点寄件人
- 关注顺序
- 自定义关注内容
- 已有板块规则
- 历史反馈记录
- 线程上下文
- 聚合逻辑结果
- 当前前端交互状态中对判断有帮助的部分

换句话说，系统不是“只看 Gmail 原文”，而是“结合 Gmail 数据与本地偏好状态共同分析”。

---

## 五、LLM 在系统中的作用

### 1. LLM 的职责

LLM 的职责不是写一段自由文本解释，而是把每封邮件转成：

- 一组 **稳定、可比较、可复用的结构化特征**
- 一组 **直接供前端展示的摘要字段**

### 2. LLM 输出内容分为两类

#### A. 模型特征输出

这些特征供后续规则层与本地模型使用，例如：

- 是否存在明确 DDL
- 是否 48h 内需要查阅
- 是否 48h 内需要处理
- 是否需要回复
- 是否需要提交材料
- 是否需要下载附件
- 是否仅供知晓
- 是否广告 / 推广 / 订阅 / 群发
- 是否命中用户关注方向
- 是否来自重点寄件人
- 是否匹配自定义关注内容
- 主题标签（学业 / 招聘 / 工作 / 财务 / 面试 / 会议等）

#### B. 前端展示输出

这些字段直接供邮件卡片展示：

- `displaySender`
- `briefSubject`
- `aiSummary`

其中：

- `displaySender`：概括来源主体是谁、什么组织或什么 App
- `briefSubject`：用中文高度概括邮件主旨的短语
- `aiSummary`：一句话概括邮件真正说了什么、包含什么关键信息

### 3. 为什么展示字段也由 LLM 第一次产出

因为如果让前端展示摘要和模型特征走两套不同链路，会出现：

- 前端看到的内容和模型理解不一致
- 同一封邮件被重复调用 LLM
- 成本更高，且结果不稳定

所以本项目采用：

> **第一次 LLM 分析同时输出“模型特征 + 前端展示字段”**

## 六、本地优先级模型是如何建立的

本项目的本地模型目标是学习：

> 用户心里觉得“这封邮件有多重要”

这个问题本质上不是板块问题，而是一个 **优先级评分问题**。

### 1. 输入变量

设：

- `X`：邮件的特征向量
- `β`：模型参数
- `Z = sigmoid(β^T X)`：邮件优先级分数

则：

- `Z ∈ [0,1]`
- `Z` 越大，表示邮件越重要

### 2. 为什么选这种形式

因为这种线性可解释模型有几个优点：

- 可以直接看出每个特征对分数的影响方向
- 容易结合用户偏好做权重约束
- 小样本下更稳定
- 训练和更新成本低，适合本地快速迭代

### 3. 优先级等级映射

模型先输出连续分数 `priority_score`，然后通过阈值映射成：

- `high`
- `medium`
- `low`
- `ignore`

例如可以设计成：

- `0.75 ~ 1.00 → high`
- `0.50 ~ 0.75 → medium`
- `0.25 ~ 0.50 → low`
- `0.00 ~ 0.25 → ignore`

阈值本身也可以随着反馈进行调整。

### 4. 初始 beta 的设计

初始 `β` 并不是随机的，而是根据业务语义进行合理初始化。  
例如：

- 明确 DDL、明确 action item、重点寄件人、用户高关注方向，这些特征初始权重较高
- 广告、营销、订阅、低价值通知，这些特征初始权重较低甚至为负

这样可以让系统在第一次运行时就具备较合理的排序能力，而不是完全依赖用户后续纠偏。

### 5. 用户偏好如何进入模型

如果用户设置关注顺序为：

1. 学业
2. 招聘
3. 工作
4. 财务

则模型会尽量保持：

```text
weight(学业) > weight(招聘) > weight(工作) > weight(财务)
```

如果用户增加了自定义关注内容，则系统会：

- 把该内容加入 LLM 上下文
- 增加对应特征维度
- 在后续训练和推理中使用该维度

## 七、板块归属逻辑是如何建立的

板块不是由优先级直接决定，而是由独立逻辑判断。

### 1. 基础原则

首先，只有 **非 ignore** 邮件才允许进入：

- 优先内容
- 48h
- 待处理

如果一封邮件已经被判为 `ignore`，它应直接进入“可忽略”。

### 2. 板块逻辑

#### A. 可忽略

满足以下任一倾向：

- 低价值通知
- 广告 / 推广 / 订阅 / 群发
- 没有行动价值且优先级已落入 ignore 区间

则进入：

- `ignore`

#### B. 48h

邮件在语义上属于“48 小时内需要查看或处理”，例如：

- 明确近期截止
- 明确 48h 内事件
- 即将面试 / 会议 / 提交

则进入：

- `within_48h`

#### C. 待处理

邮件需要用户手动处理，但不一定卡在 48h 内，例如：

- 需要回复
- 需要提交材料
- 需要确认信息
- 需要补充步骤

则进入：

- `todo`

#### D. 优先内容

当一封邮件满足：

- 不是 `ignore`
- 不是 `within_48h`
- 不是 `todo`

但整体重要性仍然较高时，进入：

- `priority_content`

可以把它理解成三圆相交逻辑外的“高价值剩余区”。

## 八、前端是如何消费这些结果的

前端不会自己再重新推理邮件，而是直接消费后端输出的结构化结果。

### 1. 前端拿到的关键字段

每封邮件至少包含：

- `board_type`
- `priority_score`
- `priority_level`
- `displaySender`
- `briefSubject`
- `aiSummary`

### 2. 前端展示方式

- 按 `board_type` 进入不同导航板块
- 同板块内按 `priority_score` 从高到低排序
- 卡片左侧用分数 / 颜色体现优先级
- 卡片正文展示来源、概括主题和一句话总结
- 相似邮件可聚合显示，再由用户展开查看

### 3. 已完成逻辑

前端支持用户把邮件标记为“已完成”，其流程是：

- 用户点击“已完成”
- 邮件短暂保留撤销时间窗口
- 之后进入“24h内已完成”板块
- 用户可恢复为未完成，回到原始板块

这个流程不改变邮件的原始优先级模型，只是改变当前展示状态。

## 九、用户反馈机制

系统目前保留两个核心反馈按钮：

- 更重要
- 少显示

### 1. 更重要

表示用户认为系统低估了该邮件的重要程度。  
这会影响后续优先级训练目标，使该邮件在后续训练中被拉向更高优先级。

### 2. 少显示

表示用户认为系统高估了该邮件的重要程度，或者用户不希望它频繁出现在显眼位置。  
这会影响后续训练目标，使该邮件在后续训练中被拉向更低优先级。

### 3. 反馈如何生效

反馈不会立即粗暴修改所有结果，而是：

```text
用户点击反馈
→ 写入本地反馈数据
→ 用户点击“重新分析”
→ 重新跑 LLM 特征抽取 / 读取反馈样本
→ 更新模型参数 β
→ 重新输出 priority_score / priority_level / board_type
```

这样做可以保证：

- 逻辑稳定
- 有训练闭环
- 不会因为单次点击破坏业务语义

---

## 十、重新概述功能

“重新概述”是一个单独的功能，它的作用只有一个：

> 重新生成当前邮件在前端展示用的概括内容

它只更新：

- `displaySender`
- `briefSubject`
- `aiSummary`

它不会修改：

- `board_type`
- `priority_score`
- `priority_level`

这样就能在“不影响模型结果”的前提下，单独提高摘要质量。

## 十二、如何运行项目

### 1. 安装前端依赖

```bash
npm install
```

### 2. 安装后端依赖

```bash
pip install -r backend/requirements.txt
```

### 3. 配置环境变量

复制 `.env.example` 为 `.env`，并填写：

- Gmail 相关配置
- OpenAI API Key 或可用的本地 LLM 配置

### 4. 启动后端

```bash
python -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8010
```

健康检查地址：

[http://127.0.0.1:8010/health](http://127.0.0.1:8010/health)

### 5. 构建扩展

```bash
npm run build:extension
```

### 6. 加载扩展

1. 打开 `chrome://extensions`
2. 开启开发者模式
3. 点击“加载已解压的扩展程序”
4. 选择构建好的扩展目录
5. 打开 Gmail 页面进行使用
