# 邮件读取与内容处理的隐私安全方案

本文档给出当前项目的隐私保护边界，以及推荐的工程治理方案。

## 1. 数据流原则

- 默认最小权限：仅申请 `gmail.readonly`
- 默认最小数据：只拉取用于当前分析所需的邮件字段
- 默认最小保留：不保留不必要的原始邮件正文
- 默认最小暴露：只有分析链路会接触邮件内容

## 2. 数据是否上传

当前架构中，邮件内容会经过以下路径：

1. Gmail 页面内的 Chrome 扩展读取邮件摘要与正文片段
2. 扩展将必要字段发送到本地 FastAPI
3. FastAPI 根据配置选择：
   - 本地规则引擎
   - 本地 Ollama
   - 第三方 LLM API

因此应明确向用户说明：

- 若使用规则引擎：邮件内容不离开本机
- 若使用本地 Ollama：邮件内容仍只在本机处理
- 若使用云端 LLM：参与分析的邮件内容会发送给配置的模型服务商

## 3. 存储策略

推荐策略如下：

- `chrome.storage.local` 只保存：
  - 分析结果摘要
  - 用户偏好
  - 反馈记录
  - 已处理状态
- 不长期保存完整原始邮件正文
- 后端默认不落盘原始邮件正文
- 若必须调试，调试日志默认关闭正文打印

## 4. 加密与传输

### 4.1 传输

- 扩展到本地 FastAPI：本机回环地址通信
- 若未来部署到远程服务，必须使用 HTTPS/TLS
- 第三方 LLM API 必须使用 HTTPS

### 4.2 静态数据

推荐：

- 本地偏好与缓存采用系统级磁盘加密保护
- 若引入服务端数据库，应对敏感字段启用静态加密
- API Key 不写入前端，不进入扩展脚本，不写死在仓库

## 5. 权限控制

- Chrome 扩展仅使用 `gmail.readonly`
- 不申请 `gmail.modify` 或发信权限，避免意外写操作
- 后端只暴露必要 API：
  - `/health`
  - `/api/v1/analyze/messages`
  - `/api/v1/preferences`
- 生产化后应增加：
  - 用户身份鉴权
  - API 访问令牌
  - 请求速率限制

## 6. 内容使用范围

邮件内容仅可用于：

- 邮件摘要生成
- 优先级判断
- 待处理/到期/可忽略分类
- 用户反馈驱动的排序优化

邮件内容不得用于：

- 广告营销
- 无关训练数据沉淀
- 未经授权的人工浏览
- 与当前用户无关的画像构建

## 7. 防泄露与防滥用措施

### 7.1 代码与日志层

- 禁止在日志中打印完整邮件正文
- 错误日志默认脱敏，只保留 messageId、发件人域名、状态码等必要信息
- 调试模式与生产模式分离

### 7.2 配置层

- `.env` 不入库
- API Key 定期轮换
- 第三方模型服务地址白名单化

### 7.3 运维层

- 若未来部署服务端，需开启审计日志
- 管理员与开发者访问最小授权
- 敏感数据导出必须可审计

### 7.4 产品层

- 首次授权时明确说明数据流向
- 允许用户选择：
  - 仅本地规则分析
  - 本地模型分析
  - 云端模型分析
- 允许用户清除本地分析缓存和偏好记录

## 8. 推荐的默认安全模式

对隐私要求高的用户，推荐默认模式为：

1. Gmail 扩展读取邮件
2. 本地 FastAPI 处理
3. 优先使用规则引擎或本地 Ollama
4. 仅在用户主动开启时才允许发送到云端 LLM

## 9. 后续可继续加强的方向

- 对缓存摘要增加本地加密
- 增加“敏感字段脱敏后再送模型”的预处理
- 为不同发件人/主题设置更细粒度的数据保留时间
- 增加“仅分析标题与摘要，不上传正文片段”的隐私模式
