# 优先级建模与排序规则

本文档只定义“优先级”这条链路，不定义板块归属。板块归属和优先级必须解耦：

- `board_type` 决定邮件在前端哪个板块展示
- `priority_score / priority_level` 决定邮件本身有多重要

## 1. 优先级模型目标

优先级模型学习的是用户对邮件重要程度的心理预期，而不是邮件该放到哪个板块。

定义如下：

```text
X = 邮件特征向量
β = 模型参数
Z = sigmoid(β^T X)
```

其中：

- `X` 为结构化特征
- `β` 为特征权重
- `Z ∈ [0,1]`

再通过阈值把 `Z` 映射为：

- `high`
- `medium`
- `low`
- `ignore`

## 2. 输入来源

优先级模型的输入不能只来自 Gmail 文本，还应同时吸收项目内已有上下文：

- Gmail 规范化字段
- 用户偏好设置
- 重点寄件人
- 已选关注标签
- 标签顺序
- 自定义关注内容
- 历史点击与历史反馈
- 线程上下文
- 已有排序和已处理状态

## 3. LLM 结构化特征

LLM 的职责是把邮件转成固定 schema 的可计算特征，而不是输出自由文本结论。

### 3.1 基础特征类别

至少应覆盖：

- 时间与 DDL 特征
- 内容主题特征
- 操作需求特征
- 干扰类特征
- 用户偏好命中特征

### 3.2 允许扩展

特征数量不设死限制，只要新增特征满足以下条件即可加入模型：

- 含义明确
- 输出稳定
- 对板块判断或优先级判断有用
- 不与已有特征严重重复

如果用户新增自定义关注内容，则必须同步做到：

1. 在偏好设置中记录
2. 进入 LLM 输入上下文
3. 扩展 LLM 输出 schema
4. 在特征向量中新增对应维度
5. 进入训练与推理流程

## 4. 偏好权重约束

优先级模型必须体现用户偏好顺序。

例如用户设置：

1. 学业
2. 招聘

则模型需要保证：

- 学业相关特征权重 > 招聘相关特征权重
- 两者都高于普通 tag 权重

同理，重点寄件人命中、历史显式提升反馈等信号，也应通过权重进入 `β`。

## 5. 阈值与档位

建议将 `Z` 用若干阈值映射到 4 档：

- 高优先级
- 中优先级
- 低优先级
- 可忽略级

阈值本身也属于可优化参数，不应写死在产品逻辑里。随着反馈样本积累，系统应允许同时优化：

- `β`
- 优先级阈值分段

## 6. 同板块内排序

板块内部排序主要参考：

1. `priority_score`
2. 时间相关特征
3. 是否未读
4. 是否命中重点寄件人
5. 当前项目已有排序规则

这意味着：

- 48h 板块里也可以出现不同优先级邮件
- 待处理板块里也可以出现不同优先级邮件
- 板块不会替代优先级，优先级也不会替代板块

## 7. 反馈如何修正优先级

当前有效反馈按钮只有两类：

- `更重要`
- `少显示`

反馈默认优先作用于优先级模型：

- `更重要`：提高该邮件的目标优先级
- `少显示`：降低该邮件的目标优先级

这里不建议把一次点击直接解释为板块迁移，否则会把 48h / 待处理这类业务语义打乱。

## 8. 重训流程

当用户点击“重新分析”时，优先级链路应执行：

1. 读取 Gmail 最新数据
2. 读取当前项目已有设置
3. 重跑 LLM 特征抽取
4. 读取反馈记录
5. 更新训练样本
6. 优化 `β`
7. 优化阈值
8. 重新输出 `priority_score / priority_level`

优化目标可以是：

- 最大化优先级分类正确率
- 最小化预测优先级与用户反馈目标之间的误差
- 对高价值邮件给予更高误判惩罚

可选方法包括：

- 梯度下降
- 随机梯度下降
- 带正则项的线性优化

## 9. 结果约束

优先级链路最终必须稳定输出：

- `priority_score`
- `priority_level`
- 用于展示的优先级原因标签

并保证：

- 同一邮件在相同输入下可重复生成
- 同一特征在不同邮件之间具备统一判断标准
- 前端颜色只读取 `priority_level`
